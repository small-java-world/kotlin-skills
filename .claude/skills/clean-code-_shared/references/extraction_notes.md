# エクストラクションノート

このファイルは、保守性に関するガイダンスをスキルルールに正規化した経緯を記録している。
逐語引用ではなく、一般的なソフトウェア工学の原則を蒸留・再構成したマッピングである。

## ソースの参照先

- ソフトウェア保守性・クリーンコードに関する一般的な知識体系
  - SOLID原則、リファクタリング手法、テスト設計パターン等の公知の概念を統合
  - プロジェクト固有の実践ルールとして20ルール体系（CC-P/C/T/K）に再構成

## 正規化されたテーマ

1. **保守性が重要な理由**
   - 継続的な変更速度は保守性に依存する。
   - 技術的負債による短期的なスピードアップは、長期的な開発速度を下げる。

2. **運用可能な原則**
   - KISS・YAGNI・DRY・SRP・CQS を実践的なレビュールールとして使う。
   - 修正は小さく、振る舞いを保持するものにする。

3. **変更戦略**
   - 振る舞いの変更と構造的リファクタリングを分離する。
   - インクリメンタルなマイグレーションと影響マッピングを活用する。

4. **テスト戦略**
   - リスクに応じて単体/結合/E2E を組み合わせる。
   - 明確な名前とアサーションを持つ決定論的なテストを優先する。

5. **チームプロセス**
   - レビューは根拠（evidence）ベースで行う。
   - 指摘には必ず根拠と具体的な検証手順を含める。

## 意図的なスコープ外

以下のトピックは評価したが、スキルルールから意図的に除外している。
人間の判断・プロジェクト固有の慣習が必要、またはコードレベルのレビューの範囲を超えるため。

| トピック | 除外理由 |
|---|---|
| 命名・コメント・ドメイン表現 | 命名品質はコンテキスト依存だが、曖昧命名による誤読リスクが高いため `CC-P006`（Naming & Domain Language）としてルール化済み。 |
| コードの分類術（ディレクトリ/モジュール構成） | チームごとに慣習が異なる。CC-P004（SRP）が部分的にカバー。 |
| アーキテクチャ（レイヤード/クリーン/MVVM） | アーキテクチャ判断にはシステム全体のコンテキストが必要でスニペットレビューの範囲外。`anti_patterns.md` の「漏れる境界」が部分的にカバー。 |
| 車輪の再発明・銀の弾丸 | 一般的なエンジニアリングの知恵であり、コードレビュールールとして自動化できない。 |
| Git/コードレビュー/設計書 | プロセスとツールのガイダンス。`output_contract.md` と `anti_patterns.md` の「レビュー品質」でカバー済み。 |

## 反映済み拡張

- **CC-P001 の拡張**: プリミティブ型の濫用と不要なミュータブル状態を KISS の具体スメルとして追加済み。
- **CC-C002 の拡張**: 結合度6レベル（内容/共通/外部/制御/スタンプ/データ結合）を診断語彙として追加済み。
- **CC-P004 の拡張**: 凝集度レベル（偶発的→機能的）を SRP 診断語彙として追加済み。
- **CC-P006 の追加**: 命名とドメイン語彙の一貫性を独立ルール化。
- **CC-T005 の追加**: 時刻・乱数・スレッド・スケジューラ等の非決定境界を独立ルール化。
- **CC-K001〜K005 の追加**: Kotlin 固有ルール（構造化並行性・Flow誤用・Null境界・スコープ関数・Data class）を新 CC-K プレフィックスとして追加。

## クロスカッティングルールの所属判断基準

CC-K ルールは principles-architect に配置し、change-safety-reviewer / testability-optimizer には配置しない。

| ルール | 所属 | 判断理由 |
|--------|------|---------|
| CC-K001（構造化並行性） | principles-architect | CoroutineScope のライフサイクルは「設計原則」レベルの判断。CC-C002（隠れ結合）と関連するが、根本原因の検出は principles 側で行い、変更安全性への波及は CC-C002 が自然にカバーする |
| CC-K002（Flow/Channel） | principles-architect | Flow の誤用は CQS（CC-P005）の Kotlin 固有表現。reactive パターンの選択は原則レベル |
| CC-K003（Null境界） | principles-architect | null 安全は Kotlin の型システム設計。テスタビリティ（CC-T005）は結果的影響であり、根本は型設計の問題 |
| CC-K004（スコープ関数） | principles-architect | 可読性・KISS（CC-P001）の Kotlin 固有表現 |
| CC-K005（Data class） | principles-architect | 等価性契約は設計原則。変更安全性への影響は CC-C002 が検出 |

**原則**: 根本原因を検出するエージェントが所有する。波及効果は他エージェントの既存ルールがカバーする。
