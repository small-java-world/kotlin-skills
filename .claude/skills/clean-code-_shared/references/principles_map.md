# 原則マップ

## KISS (`CC-P001`)
- **意図**: 実装を安全に変更できる最小の複雑さに抑える。
- **スメル**:
1. 不要な間接参照レイヤー（抽象化のためだけの抽象化）
2. 過剰なパラメータを持つユーティリティラッパー
3. 固定動作なのに設定だけが膨らんでいる
4. プリミティブ型の濫用：ドメイン型にすれば不正状態を排除できるのに `Int`/`String`/`Boolean` をそのまま使っている
5. 不要なミュータブル状態：イミュータブルで十分なスコープで `var` や `mutableListOf` を使っている
- **最小修正パターン**:
1. 抽象化レイヤーを1つインライン化する
2. 決定を呼び出し元に移すか、使われていない選択肢を削除する
3. 一本道のコードパスを保つ
4. 値ドメインを制約するために名前付き型（inline class / value class / enum）を導入する
5. `var` を `val` に置き換え；構築時のミュータビリティは `buildList {}` 等で対応

## YAGNI (`CC-P002`)
- **意図**: 現在の要件が必要とするものだけを実装する。
- **スメル**:
1. 使われていない拡張ポイント
2. 投機的なプラグインシステム
3. 一度も消費されない休眠フラグ
- **最小修正パターン**:
1. 使われていないブランチやインターフェースを削除する
2. 明確なトリガー条件付きの TODO として残す

## DRY (`CC-P003`)
- **意図**: 知識の重複を減らす（構文的な繰り返しではなく）。
- **スメル**:
1. 同じビジネスルールが複数箇所にコピーされている
2. バリデーションロジックが重複しており、片方だけ変更されてズレが生じている
- **最小修正パターン**:
1. ルールを1つの関数/モジュールに集約する
2. 呼び出し側は読みやすさのために明示的なままにする

## SRP (`CC-P004`)
- **意図**: 各モジュールが変更される理由を1つにする。
- **スメル**:
1. ドメインロジックがトランスポート/永続化と混在している
2. クラスがポリシー＋フォーマット＋I/O を一緒に扱っている
3. 偶発的凝集：「同じタイミングで動く」という理由だけでまとめられた操作（関連がない）
4. 論理的凝集：1つの関数内でフラグ/enum パラメータが無関係な動作を切り替えている
- **最小修正パターン**:
1. ポリシーを I/O から分離する
2. オーケストレーションを計算から分離する
3. 偶発的なグループ化を、目的を表す名前の関数（1文で言い表せる）に置き換える
4. フラグで分岐している関数を、明確な名前を持つ2つの関数に分割する
- **凝集度参照（低 → 高）**: 偶発的 → 論理的 → 時間的 → 手続き的 → 通信的 → 逐次的 → 機能的
  - 偶発的/論理的：リファクタ優先度が高い
  - 時間的〜通信的：許容できるが肥大化に注意
  - 逐次的/機能的：目標状態

## 結合度レベル（CC-C002 の診断語彙）
隠れた結合を診断するときに使う。最も有害なものから順に並べている：
1. **内容結合**（Content）：あるモジュールが別のモジュールの内部を直接読み書きしている
2. **共通結合**（Common）：2つのモジュールがグローバルなミュータブル状態を共有している
3. **外部結合**（External）：どちらも所有しない外部フォーマット/プロトコルを共有している
4. **制御結合**（Control）：呼び出し元が被呼び出し側の動作を制御するフラグを渡している（CC-P004 の論理的凝集と重複）
5. **スタンプ結合**（Stamp）：1フィールドしか必要ないのにデータ構造全体を渡している
6. **データ結合**（Data）：必要なデータだけを渡している — 許容レベル
- **CC-C002 のスメル**: レベル1〜4 は隠れた結合を示す。被呼び出し側がシグネチャから分かる以上に呼び出し元の意図を知っているときに警告する。

## CQS (`CC-P005`)
- **意図**: 状態変更とクエリを分離して予期しない挙動を防ぐ。
- **スメル**:
1. 副作用を持つ getter
2. 古いキャッシュデータを返すコマンドメソッド
- **最小修正パターン**:
1. コマンドとクエリのメソッドに分割する
2. 副作用が明確に分かるようにメソッドを改名する

## Naming & Domain Language (`CC-P006`)
- **意図**: 意味のある命名で意図と境界を明示し、誤用を防ぐ。
- **スメル**:
1. `process`, `handle`, `data`, `flag` のような曖昧な名前に責務が隠れている
2. ドメイン語彙が揺れていて同じ概念を別名で表している
3. Kotlin の Boolean フラグ引数（`isX`, `doY`）で無関係な振る舞いを切り替えている
- **最小修正パターン**:
1. 名前を「何を保証するか」が分かるドメイン語彙に置き換える
2. 揺れている語彙を型名/関数名で1つに統一する
3. フラグ引数を目的別の明示的メソッドに分割する

## Determinism Boundary (`CC-T005`)
- **意図**: 時刻・乱数・スレッド・スケジューラなど非決定要素を境界化してテスト可能にする。
- **スメル**:
1. `Instant.now()` / `LocalDateTime.now()` / `Random.next*()` を直接呼ぶ
2. `Dispatchers.IO` / `GlobalScope` / 固定 `delay` を本番コードにハードコードする
3. テストが `Thread.sleep` 依存になり、再現性のない失敗が出る
- **最小修正パターン**:
1. Clock / Random / Dispatcher / Scope を注入して差し替え可能にする
2. 時間待ちを scheduler 制御に置き換え、テストで `advanceUntilIdle` を使う
3. 非決定境界を1か所に集約して単体テストでは fake 実装を使う

## 構造化並行性違反 (`CC-K001`)
- **意図**: coroutine の生存期間を呼び出し元スコープに紐付け、リークと例外の取りこぼしを防ぐ。
- **スメル**:
1. GlobalScope.launch / GlobalScope.async を本番コードで使用
2. CoroutineScope を外部から渡さず内部で生成（スコープが漏れる）
3. launch { } 内の例外を supervisorScope や CoroutineExceptionHandler で拾わず握りつぶしている
- **最小修正パターン**:
1. GlobalScope を注入された CoroutineScope（またはライフサイクルスコープ）に差し替える
2. launch → async に切り替え await() で例外を伝搬させる
3. 例外ハンドラを CoroutineScope の CoroutineExceptionHandler として登録する

## Flow/Channel 誤用 (`CC-K002`)
- **意図**: cold/hot の区別を明示し、状態の共有・購読の副作用を制御する。
- **スメル**:
1. flow {} (cold) を StateFlow/SharedFlow (hot) と混同して使用し、重複実行や状態消失が起きる
2. SharedFlow の replay/extraBufferCapacity を 0 のまま使い、遅い購読者がイベントを取りこぼす
3. collect {} ブロック内で外部状態を直接変更する副作用コード
- **最小修正パターン**:
1. 状態保持には StateFlow/MutableStateFlow、イベントバスには SharedFlow(replay=1+) を使い分ける
2. replay と onBufferOverflow を意図に合わせて明示設定する
3. collect 内の副作用を onEach / launchIn に移しフローを純粋に保つ

## Null 安全性の境界漏れ (`CC-K003`)
- **意図**: Kotlin の null 安全を Java/外部 API との境界で失わない。
- **スメル**:
1. Java ライブラリの戻り値をプラットフォーム型（!型）のまま受け取り、null チェックなしで使用
2. lateinit var を null 安全の代替として濫用し、UninitializedPropertyAccessException のリスクを隠蔽
3. !! 以外の形（as 強制キャスト、Java interop の暗黙型推論）で null 安全を突破している箇所
- **最小修正パターン**:
1. Java API の戻り値は ?: エルビス演算子か requireNotNull/checkNotNull で境界処理する
2. lateinit は DI フレームワークが注入するフィールドのみに限定し、用途をコメントで明示する
3. プラットフォーム型には明示的な nullable 型注釈（Type?）を付けて意図を文書化する

## スコープ関数の過剰チェーン (`CC-K004`)
- **意図**: let/run/apply/also/with の連鎖を最小限に抑え、可読性と debuggability を保つ。
- **スメル**:
1. 3つ以上のスコープ関数をチェーンして1式に詰め込んでいる
2. it と this が混在し、内側スコープでどのレシーバを指しているか分からない
3. スコープ関数でサイドエフェクトとデータ変換の両方を同時に行っている
- **最小修正パターン**:
1. チェーンを中間変数付きの逐次ステップに展開する（2ステップ以内に分割）
2. 変換には let/run、設定には apply/also と用途を1種に絞る
3. サイドエフェクトと変換を別の式に分離し、also を最後にまとめる

## Data class 契約違反 (`CC-K005`)
- **意図**: data class の equals/hashCode/copy の契約を保ち、予期しない比較・コピー動作を防ぐ。
- **スメル**:
1. data class のプロパティに var（ミュータブル）を使用し、コレクション格納後に値が変化する
2. data class を継承して equals/hashCode が上書きされ、親子間の比較が非対称になる
3. copy() で一部プロパティだけ変更した際、コンパニオンやバリデーションをバイパスしている
- **最小修正パターン**:
1. var を val に変更し、更新は copy() を使って新インスタンスを生成する
2. data class の継承をやめ、sealed class / sealed interface で代替する
3. ファクトリ関数（companion object.of()）でバリデーションを一元化し copy() を隠蔽する
