# アンチパターン

## アーキテクチャと設計
1. **Big-ball モジュール**: 無関係な責務が1つのファイル/クラスに詰め込まれている。
2. **ユーティリティの墓場**: グローバルなヘルパーが所有者なしに肥大化し続けている。
3. **漏れる境界**: ドメインロジックがフレームワークの詳細に依存している。
4. **曖昧命名の連鎖**: `data/process/flag` のような名前が連鎖し、意図と責務境界が読めない。

## 変更安全性
1. **[CC-C003] 混合変更意図**: 振る舞いの変更と構造的リファクタリングを1つのコミットで行っている。
2. **[CC-C001/C002] 継ぎ目なしの広範囲編集**: 中間の安全レイヤーなしに多くのモジュールを同時に変更している。
3. **[CC-C002] 隠れた依存**: 関数シグネチャから副作用が見えない。
4. **[CC-C001] 単一変更への過剰な影響範囲**: 1回のコミット/PRが複数モジュール・データパスにわたり、レビュアーが全影響を把握できない。段階分割を検討する。
5. **[CC-C004] マイグレーションパスなしのリファクタリング**: 大規模な構造変更を1ステップで行い、中間に安全網となるコミットや段階的な計画がない。

## テスタビリティ
1. **実装詳細への結合**: テストが内部実装に依存しており、リファクタリングで壊れる。
2. **全 E2E 戦略**: 単体/結合テストの代わりに遅い E2E テストだけで網羅しようとしている。
3. **振る舞いを表さないアサーション**: ユーザーが見える動作ではなく実装の詳細を検証している。
4. **モックの過剰使用**: テスト内で使っても安全・高速な協調クラスまで偽物に差し替えている。これにより実際の統合バグを検出できなくなる。依存先が遅い・非決定論的・外部インフラを必要とする場合のみ偽物を使うこと。
5. **動的な期待値**: テストコード内でプロダクションコードと同じロジックで期待値を計算している。テストが循環してしまい、丸め/切り捨てエラーを検出できなくなる。期待値にはハードコードした定数を使うこと。
6. **AAA の複数サイクル**: 1つのテストケースで Arrange→Act→Assert を複数回繰り返しており、どのシナリオが失敗したか分からなくなる。シナリオごとに1つのテストに分割すること。
7. **非決定要素の直結**: 時刻・乱数・ディスパッチャ・待機時間を直接呼び出し、テストが時間依存で不安定になる。

## レビュー品質
1. 根拠（evidence）のない指摘。
2. 最小修正提案のない指摘。
3. 曖昧な言語を使っている指摘（"somehow", "maybe", "as needed" など）。

## Kotlin 固有
1. **[CC-K004] スコープ関数の連鎖地獄**: `let`/`run`/`apply`/`also`/`with` を3段以上チェーンし、レシーバの追跡が不可能になっている。
2. **[CC-K001] 非構造化 coroutine の生成**: `GlobalScope` や内部生成した `CoroutineScope` を使い、coroutine の生存期間がライフサイクルに紐付かずキャンセルや例外が伝搬しない。
3. **[CC-K002] Flow の cold/hot 混同・暗黙ドロップ**: `SharedFlow`/`StateFlow` を cold フローとして扱う、`collect {}` 内で副作用を実行して重複実行が起きる、`conflate()` で中間値を暗黙に捨てる。
4. **[CC-K003] Null 安全の境界漏れ**: Java 由来の platform type を non-null として信頼する、`as` でキャストして `NullPointerException` を隠す、`lateinit var` を nullable の代替として濫用する。
5. **[CC-K005] data class 契約違反**: `var` プロパティを持つ data class をコレクションのキーに使用する（ミュータブルキー）、`open data class` を継承して `equals`/`hashCode` 非対称を引き起こす、`companion object` ファクトリのバリデーションを `copy()` が迂回する。
6. **companion object の責務肥大化** *(ルールなし — 一般的な Kotlin スメル)*: companion object がファクトリ以外の複数責務（バリデーション・キャッシュ・設定）を持ち、単一クラスの巨大ユーティリティと化している。
7. **object シングルトンのテスト困難性** *(ルールなし — 一般的な Kotlin スメル)*: `object` 宣言で状態を持つシングルトンを作り、テスト間でグローバル状態が汚染される。
8. **init ブロックでの副作用** *(ルールなし — 一般的な Kotlin スメル)*: `init {}` ブロックで I/O・DBアクセス・coroutine 起動などの副作用を行い、インスタンス生成の制御が困難になっている。
