# エクストラクションノート

このファイルは、保守性に関するガイダンスをスキルルールに正規化した経緯を記録している。
逐語引用ではなく、一般的なソフトウェア工学の原則を蒸留・再構成したマッピングである。

## ソースの参照先

- ソフトウェア保守性・クリーンコードに関する一般的な知識体系
  - SOLID原則、リファクタリング手法、テスト設計パターン等の公知の概念を統合
  - プロジェクト固有の実践ルールとして13ルール体系（CC-P/C/T）に再構成

## 正規化されたテーマ

1. **保守性が重要な理由**
   - 継続的な変更速度は保守性に依存する。
   - 技術的負債による短期的なスピードアップは、長期的な開発速度を下げる。

2. **運用可能な原則**
   - KISS・YAGNI・DRY・SRP・CQS を実践的なレビュールールとして使う。
   - 修正は小さく、振る舞いを保持するものにする。

3. **変更戦略**
   - 振る舞いの変更と構造的リファクタリングを分離する。
   - インクリメンタルなマイグレーションと影響マッピングを活用する。

4. **テスト戦略**
   - リスクに応じて単体/結合/E2E を組み合わせる。
   - 明確な名前とアサーションを持つ決定論的なテストを優先する。

5. **チームプロセス**
   - レビューは根拠（evidence）ベースで行う。
   - 指摘には必ず根拠と具体的な検証手順を含める。

## 意図的なスコープ外

以下のトピックは評価したが、スキルルールから意図的に除外している。
人間の判断・プロジェクト固有の慣習が必要、またはコードレベルのレビューの範囲を超えるため。

| トピック | 除外理由 |
|---|---|
| 命名・コメント・ドメイン表現 | 命名品質はコンテキスト依存で信頼性のある自動ルール化が困難。将来の候補: `CC-P006`。 |
| コードの分類術（ディレクトリ/モジュール構成） | チームごとに慣習が異なる。CC-P004（SRP）が部分的にカバー。 |
| アーキテクチャ（レイヤード/クリーン/MVVM） | アーキテクチャ判断にはシステム全体のコンテキストが必要でスニペットレビューの範囲外。`anti_patterns.md` の「漏れる境界」が部分的にカバー。 |
| 車輪の再発明・銀の弾丸 | 一般的なエンジニアリングの知恵であり、コードレビュールールとして自動化できない。 |
| Git/コードレビュー/設計書 | プロセスとツールのガイダンス。`output_contract.md` と `anti_patterns.md` の「レビュー品質」でカバー済み。 |
